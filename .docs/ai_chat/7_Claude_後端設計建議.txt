⏺ Here are three approaches:

  ---
  Option A: Supabase (my recommendation)
  Hosted Postgres + built-in auth (email/password + Google OAuth). Stock
  transaction data is naturally relational, so CSV → Postgres tables is a clean
  migration. Row-level security (RLS) handles per-user data isolation elegantly.
   Open source — can self-host later. Free tier, no server to manage.

  Option B: Firebase (Auth + Firestore)
  Firebase Auth handles email/password + OAuth out of the box. Firestore is
  NoSQL, which is awkward for tabular stock/transaction data — you'd fight the
  data model. Google vendor lock-in. Easier auth setup, harder data modeling.

  Option C: SQLite + JWT (self-managed)
  Keep data local in SQLite (one DB, per-user rows), add JWT auth yourself.
  Maximum control, zero vendor dependency. But you write all the auth
  boilerplate: password hashing, token refresh, OAuth flows. More work up front,
   and SQLite struggles under concurrent writes at scale.

  ---
  Recommendation: Option A (Supabase). It gives you production-ready auth + a
  real relational DB with minimal code, your CSV data maps cleanly to Postgres
  tables, and you're not locked in — the underlying tech is just Postgres +
  open-source tooling. Starting with just email/password auth is one config
  change away from adding Google OAuth later.

  Does this direction make sense, or do you want to explore one of the other
  options?

⏺ Section 1: Architecture

  React Frontend (Vite)
    │  1. login → Supabase Auth (email/password)
    │  2. receives JWT
    │  3. sends JWT as Bearer token on every API call

  FastAPI Backend
    │  4. verifies JWT using Supabase JWT secret → extracts user_id
    │  5. scopes all DB queries to user_id
    ▼
  Supabase
    ├─ Auth (manages users, sessions, future OAuth)
    └─ Postgres (stock/transaction data, per-user rows)

  The frontend uses the Supabase JS client only for auth. All business logic
  (holdings calculations, dashboard, etc.) stays in FastAPI — no change there.
  FastAPI gets a new auth dependency (get_current_user) that verifies the JWT

⏺ Section 2: Data Model

  Your CSV columns map cleanly to Postgres tables. Each table gets a user_id
  foreign key referencing Supabase's built-in auth.users:

  -- transactions (from transactions.csv)
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid()
  user_id     uuid REFERENCES auth.users NOT NULL
  date        date
  action      text        -- 買/賣/配息
  code        text
  name        text
  trade_type  text
  buy_shares  numeric
  buy_price   numeric
  sell_shares numeric
  sell_price  numeric
  dividend_income numeric
  reason      text

  -- cashflow (from cashflow.csv)
  id, user_id, date, action, amount, note

  -- stocks (from stocks.csv — user's watchlist)
  id, user_id, code, name

  -- user_settings (from settings.json)
  user_id     uuid PRIMARY KEY REFERENCES auth.users
  data        jsonb       -- keeps existing JSON structure

  Row-Level Security (RLS) policies on each table enforce user_id = auth.uid() —
   so even if FastAPI has a bug, Supabase won't leak another user's data.
  FastAPI's csv_service.py is replaced by supabase_service.py using the
  supabase-py client.

⏺ Section 3: Auth Flow

  Starting with email/password only. Google OAuth requires zero backend changes
  later — just enable it in Supabase dashboard.

  Login (frontend):
  import { createClient } from '@supabase/supabase-js'
  const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY)

  // sign up
  await supabase.auth.signUp({ email, password })

  // sign in — returns JWT access token
  const { data } = await supabase.auth.signInWithPassword({ email, password })
  // store data.session.access_token, attach to all API calls

  Verification (FastAPI):
  # routers/deps.py — replaces get_csv_service as the primary dependency
  async def get_current_user(token: str = Depends(oauth2_scheme)) -> str:
      payload = jwt.decode(token, SUPABASE_JWT_SECRET, algorithms=["HS256"])
      return payload["sub"]  # user_id (uuid)

  # all routers gain: user_id: str = Depends(get_current_user)

  The SUPABASE_JWT_SECRET lives in a .env file, never in code.
